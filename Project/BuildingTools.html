    构建工具发展历史

    # 构建工具的发展方向

    构建工具有两个发展方向，一个是传统的 bundle 类构建工具，需要经历打包过程；一个是近年流行的 bundleless/no-bundle 类构建工具，无需进行打包。那么为什么以前需要打包，现在不需要打包呢？

    为什么过去需要打包？

    • HTTP/1.1 各浏览器有并行连接限制
    • 浏览器不支持模块系统（如 CommonJS 包不能直接在浏览器运行）
    • 代码依赖关系与顺序管理

    为什么现在不需要打包？

    近几年时间，标准的确立、浏览器大厂和前端生态的跟进，使得「不打包」成为可能：

    • HTTP/2.0 多路并用
    • 各大浏览器逐一支持 ESM
    • 越来越多的 npm 包拥抱 ESM（尽管很多包的依赖并不是）

    # bundle 类构建工具

    1）grunt 和 gulp：

    grunt 和 gulp 比较像，它们都是基于任务的构建工具，就像进行流水线作业的自动化机器一样，避免让我们手动、重复地完成一些工作。当然两者也有区别，grunt 是配置驱动的，上手难度高，而且每次操作产生的中间态文件都会伴随着一次磁盘读写，性能不好；gulp 则是代码驱动的，每次操作产生的中间态文件会放在内存中，传递给下一次操作

    2）Browserify：

    browserify 是基于 Modules/1.x 规范的模块处理工具，致力于在浏览器端使用 CommonJS 模块

    3）Fis3：

    百度出品的构建工具，在基本的功能之上还集成了类似图片合并、资源压缩等 web 开发中所需的构建功能，而且配置简单、开箱即用，但缺点是目前不再维护

    4）Webpack：

    相比前面的工具来说，Webpack 更加面向工程化，是一款全能且主流的打包构建工具，但缺点是配置复杂、构建速度慢。针对配置复杂的问题，很多前端框架都有对应的脚手架，实现了 0 配置和开箱即用。

    5）Rollup：

    Rollup 专注于纯 JS 和 ES Module 的处理，提出了 tree-shaking 和 scope-hoisting。

    6）Parcel：

    Parcel 最大的特点在于极速、零配置。它采用了多核编译+文件缓存，构建速度极快；此外，很多功能都内置了，因此无需额外进行配置。缺点是生态不够丰富，0 配置虽方便但很难实现定制化

    7）ESbuild：

    ESbuild 是一款速度极快的构建工具，也提供了编译和压缩等功能。而它之所以这么快，主要是因为：

    1. 使用 golang 编写，天然并行；
    2. 内部算法非常精细；
    3. 没有任何第三方依赖

    # bundlelsss 类构建工具

    bundle 类构建工具的痛点：

    这类工具的核心都是从入口文件出发，递归循环地分析依赖，构建依赖图，最终把所有依赖的模块打包到一个 js bundle 中，再注入到 HTML。

    • 启动速度：因为需要进行依赖分析和打包，所以启动速度慢
    • 构建时间：项目体积越大，打包的东西就越多，构建耗时越长
    • 文件更新（HMR）：但凡有一个模块发生了改动，都需要对它的相关依赖重新进行编译构建、重新递归地构建依赖图（虽然 webpack5 实现了增量构建），这是非常耗时的
    • 生态：无敌

    bundleless 类构建工具是怎么做的：

    这类工具的核心是跳过打包这个过程，直接基于浏览器原生的 ES Module。在解析遇到导入语句的时候，会向 devServer 发起请求，devServer 对资源进行实时编译并将结果返回给浏览器。

    • 启动速度：只需要启动 devServer，其它的东西按需加载、实时编译即可。速度飞快
    • 构建时间：不会随着项目体积增大而拖慢构建时间
    • 文件更新（HMR）：每次模块发生改动的时候，可以只针对该模块进行增量构建
    • 生态：目前仍然不足以取代 Webpack，但是未来的趋势

    那么 bundleless 类构建工具有哪些？

    bundlelss 类构建工具的典型是 Snowpack 和 Vite。

    以 Vite 为例，它内置了 ESbuild 和 Rollup。Vite 使用 ESbuild 进行实时编译，指的是将 .ts、.less 等文件编译为浏览器可以解析的形式；此外，由于 Rollup 在打包上面更加成熟和灵活，所以 Vite 选择使用 Rollup 而不是 ESbuild 进行生产环境的打包。
