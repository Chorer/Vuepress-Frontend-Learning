    异步编程

    # Promise 的状态转化

    三个状态：从 pending 状态落定到 resolved/fulfilled 状态或者 rejected 状态。状态确定后不可逆。

    # Promise 的 API

    # Promise.resolve() 和 Promise.reject()

    Promise.reject() 始终会基于传入参数返回一个落定为 rejected 状态的 promise，而 Promise.resolve() 接受 promise 参数的时候，始终返回该参数；接受非 promise 参数的时候，始终返回一个落定为 resolved 状态的 promise：

    // 由于幂等性，无论包装多少层，始终等于初始传入的 promise
let p = Promise.reject()
p === Promise.resolve(Promise.resolve(p))

    # Promise.prototype.then()

    任何 promise 实例都可以调用 then 方法，根据该 promise 落定为 resolved 状态还是 rejected 状态，会决定调用 then 方法中的成功回调还是失败回调。

    1）回调接受的参数是 promise 实例 resolve 或者 reject 的值：

    Promise.resolve(1).then(res => console.log(res))       // 1
Promise.reject(2).then(null,res => console.log(res))   // 2

    2）调用 then 之后会返回一个新的 promise 实例，它会基于回调函数的返回值进行构建：

    let p = Promise.resolve(1)

// then 不接受参数，则 p1 和 p 一样
let p1 = p.then()             

// then 接受非函数参数，则静默失效，p1 和 p 一样
let p1 = p.then(123)

// then 接受函数参数，函数返回值为 promise，则 p1 和这个 promise 一样
let p1 = p.then(() => Promise.resolve(2))                // Promise <resolved>: 2
let p1 = p.then(() => Promise.reject(2))                 // Promise <rejected>: 2 
let p1 = p.then(() => Promise.resolve())                 // Promise <resolved>: undefined
let p1 = p.then(() => new Promise(() => {}))             // Promise <pending>

// then 接受函数参数，函数返回值为非 promise，则 p1 是 resolve 该返回值的 promise
let p1 = p.then(() => 123)                         // Promise <resolved>: 123
let p1 = p.then(() => new Error())                 // Promise <resolved>: Error

// then 接受函数参数，函数抛出异常，则 p1 是 reject 该异常的 promise
let p1 = p.then(() => {throw 123})                 // Promise <rejected>: 123

    PS：

    • 如果 p 是落定为 rejected 状态的 promise，则走失败回调，以上规则仍然适用
    • 如果 p 是状态未落定的 promise（pending 状态），则直接返回 Promise <pending>
    • 即使回调函数返回 Error，then 方法也仍然是返回一个 resolve 该 Error 的 promise；但如果回调函数抛出异常，则 then 会返回一个 reject 该异常的 promise
    • 链式调用空参数的 then 或者 catch，会始终返回一个相同的落定状态的 promise，从而实现值向下一层的穿透

    # Promise.prototype.catch()

    此外还有一个只由落定为 rejected 状态的 promise 调用、只走失败回调的 catch 方法。p.catch(() => {}) 相当于是 p.then(null,() => {})。

    调用 catch 方法返回的新 promise 实例同样也是基于回调函数返回值进行构建的，规则和上面一样。

    # Promise.prototype.finally()

    最后是不管 promise 落定为 resolved 状态还是 rejected 状态，都会调用的 finally 方法。大部分情况下，调用 finally 方法都会返回一个和调用者一样的 promise，但下面几种情况例外：

    let p = Promise.resolve()

// finally 接受函数参数，函数返回值为 pending/rejected 状态的 promise，则 p1 和该 promise 一样
let p1 = p.finally(() => new Promise(() => {}))           // Promise <pending>
let p1 = p.finally(() => Promise.reject(123))             // Promise <rejected>: 123

// finally 接受函数参数，函数抛出异常，则 p1 是 reject 该异常的 promise
let p1 = p.finally(() => {throw 123})                     // Promise <rejected>: 123

    # Promise 的错误冒泡和捕获

    Promise.resolve(1)
.then(res => res + 1)
.then(res => Promise.reject(res+1))        // 这里会抛出错误
.then(res => res + 1)
.catch(() => {})                           // 这里可以捕获

    第三行的代码中，回调函数的返回值是一个落定为 rejected 状态的 promise，这会抛出一个错误，并使得调用 then 之后返回的 promise 就是 Promise.reject()。这个 promise 的状态是失败的，所以不会走第四行的成功回调函数，调用第四行的 then 之后，仍然返回一个原样的 Promise.reject()，并最终被第五行的 catch 捕获。

    可以看到，then 的链式调用中任意一个环节抛出错误，这个错误都可以原路冒泡下去，并被遇到的 catch 语句捕获。

    # async 和 await

    1）为什么有了 promise 之后还需要 async 和 await？

    实际上它是为了解决异步代码组织结构的问题。假设现在有一个 promise，它会在一段时间之后 resolve 一个值。同时，有一段代码需要依赖 resolve 的这个值执行某些操作，那么使用 promise 的话，就需要这么写：

    let p = new Promise((resolve) => setTimeout(resolve,1000,'abc'))
p.then(res => {
    console.log(res)
})

    也就是说，我们必须把处理这个值的逻辑写在 then 方法的回调里。虽然可以把处理逻辑单独抽离到一个函数中，但每次只要想使用这个值，就不得不以回调函数的形式去接受它。而且，为了确保访问该值的时候它已经可用，我们也不能把它赋值给外部的一个全局变量，因为全局变量是暴露在同步逻辑中。

    如何解决这类问题呢？如果使用 async await 的话，就可以这么写：

    // chrome devtools
let p = new Promise((resolve) => setTimeout(resolve,1000,'abc'))
let res = await p

    2）await 的参数

    await 的参数如果是 promise：

    // 打印 Promise <pending>
console.log(await new Promise(() => {}))

// 打印 resolve 的值，即 123
console.log(await Promise.resolve(123))

// 直接抛出异常
console.log(await Promise.reject(123))

    await 的参数如果不是一个 thenable，则 await 会将其当作 resolve 这个参数的 promise：

    // 等价，都打印 123
console.log(await 123)
console.log(await Promise.resolve(123))

    await 的参数如果是一个非 promise 的 thenable，则会调用 thenable 的 then 方法，并且把传给 then 方法的回调函数视为 resolve 函数：

    // 等价，都打印 123
console.log(await {
    then(fn){
        fn(123)
    }
})
console.log(await new Promise(resolve => {
    resolve(123)
}))

    此外，await xxx，总是相当于用 Promise.resolve() 去包装 xxx，并将后面的逻辑都放到 then 方法的回调中：

    async function(){
    console.log(await 123)
    console.log(456)
}
// 等价于
async function(){
    Promise.resolve(123).then(res => {
        console.log(res)
        console.log(456)
    })
}

    3）async 函数的返回值

    async 函数始终期待一个 thenable 对象（具有 then 方法的对象，比如 promise）作为返回值。

    // 本身就返回 promise，则 res 为 Promise <rejected>: 123
let res = (async function(){
    return Promise.reject(123)
})()

// 返回非 promise，则 res 为 resolve 该返回值的 promise，即 Promise <resolved>: 'abc'
let res = (async function(){
    return 'abc'
})()

// 返回 thenable 对象，则 res 为该对象
let res = (async function(){
    return {
        then(fn){fn(111)}
    }
})()
res.then(console.log)           // 111

    4）异常处理

    如果 async 函数抛出异常，则会返回一个落定为 rejected 状态的 promise，并且异常可以通过 catch 捕获：

    // res 为 Promise <rejected>: 'abc'
let res = (async function(){
    throw 'abc'
    return 1                  // 已经抛出错误，因此不会执行
})()

    这等价于跟在 await 后面的 Promise.reject()：

    // res 为 Promise <rejected>: 'abc'
let res = (async function(){
    await Promise.reject('abc')
    return 1                 // 已经抛出错误，因此不会执行
})()

    注意，如果只是单独的 Promise.reject()，则仍然按照函数返回值返回 promise。同时，它抛出的异常无法被捕获：

    // res 为 Promise <resolved>: 1
let res = (async function(){
    Promise.reject('abc')
    return 1
})()

// 使用 catch 也无法捕获异常，仍然抛出
res.catch(() => {})

    此外，也可以使用传统的 try...catch 去捕获错误：

    (async function(){
    try{
        throw 123
    } 
    catch(err) {
        console.log(err)     // 123
    }
})()
(async function(){
    try{
        await Promise.reject(123)
    } 
    catch(err) {
        console.log(err)     // 123
    }
})()

    注意，这种方式同样无法捕获单独出现的 Promise.reject()。
