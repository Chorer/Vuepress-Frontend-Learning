    极客时间《浏览器工作原理与实践》学习笔记

    # 关于浏览器的架构

    # 架构的选择

    现代浏览器采用的是多进程的架构

    • 为什么不能是单线程？

      浏览器肯定要并行处理任务，即在同一段时间内做多件事情，比如渲染页面、发送请求、处理用户输入等，而单线程注定了同一时间只能做一件事情，因此不能采用单线程架构。

    • 为什么不能是单进程多线程？

      而如果采用单进程多线程架构，则 ① 如果存在恶意线程，则会影响其它正常线程，无法保证安全性；② 一个线程的崩溃会导致整个进程的崩溃，无法保证稳定性。③ 资源和数据都是共享的，无法保证互相之间的隔离

    • 为什么是多进程？

      现代浏览器采用的是类似操作系统那样的多进程架构，好处很多。进程和进程之间是互相隔离的，能够很好地保证安全性和稳定性。当然也有缺点，就是带来更高的资源占用。

    # 核心进程

    现代浏览器一般有以下几个核心进程：

    • 浏览器主进程
    • GPU 进程
    • 网络进程
    • 插件进程
    • 渲染进程

    其中，渲染进程会针对服务器返回的资源进行 HTML 解析、CSS 解析、图片解码等。如果这些资源存在问题，浏览器又存在漏洞，那么黑客就可能会控制渲染进程，进而控制操作系统。因此，为了安全起见，最好是将渲染进程单独隔离起来。

    具体地说，浏览器会分为两个内核模块，一个是渲染内核，指的是单独运行在安全沙箱中的渲染进程；一个是浏览器内核，包含了其它进程。这样，即便渲染进程出现了安全漏洞被攻破，由于沙箱的隔离作用，操作系统也不会受到影响。

    渲染进程采⽤了安全沙箱固然保证了安全，但也导致渲染进程内部不能与操作系统直接交互，于是就在浏览器内核中实现了持久存储、⽹络访问和⽤⼾交互等⼀系列与操作系统交互的功能，然后通过 IPC 和渲染进程进⾏交互。

    # 浏览器页面的渲染

    # 渲染进程

    一般来说，每打开一个标签页，就会有一个新的渲染进程；但有一种例外情况是，如果通过 a 页面打开了 b 页面，并且两个页面属于同站（same-site，即协议和根域名都一样），那么 b 页面不会重开一个渲染进程，而是会复用父页面即 a 页面的渲染进程。渲染进程包括以下6个线程：

    • GUI 渲染线程：内含渲染引擎，负责 HTML、CSS 的解析以及 DOM tree 和 CSSOM tree 的构建
    • JS 引擎线程：内含 JS 引擎，负责 JS 的解析和执行
    • 事件触发线程
    • 定时器触发线程
    • 异步 http 请求线程
    • 合成线程
    • IO 线程

    ==Note==：GUI 渲染线程和 JS 引擎线程必须是互斥的。打个比方，渲染线程在解析 HTML 准备构建 DOM 树的时候，JS 引擎线程不能同时去解析和执行 JS，否则可能在涉及 DOM 的操作上发生冲突。一定要注意，这两者不是单线程的，它们是独立的两个线程，只是互斥而已。

    # 渲染过程

    具体的渲染过程，可以参考 从输入 URL 到页面呈现的过程 (opens new window) 这篇文章。这里主要记录大致的流程和一些细节。

    # 渲染过程一览

    # 1）HTML 解析

    解析 HTML ，构建 DOM 树

    # 2）样式计算

    解析 CSS，构建 CSSOM 树（或者说 styleSheets，暂且认为是同一个东西）。之后对所有属性值进行标准化，然后基于继承和层叠规则计算每个节点的具体样式。

    # 3）布局（layout）

    • 生成布局树：DOM 树和 CSSOM 树结合，形成布局树，布局树只会包含实际显示在页面上的节点（因此不会包含 <head> 节点，隐藏的节点等）。

    • 布局计算：计算布局树中节点的具体坐标位置

    # 4）分层（layer）

    因为 z-index 和 CSS 3D 变换等的存在，页面通常不是一个简单的平面，而是有很多图层。分层指的是针对布局树中的特定节点生成属于它的图层，而多个图层会构建起一棵图层树。

    • 注意不是所有节点都会有一个属于自己的图层，像上图中的 span 就和它的父元素 div 共用一个图层
    • 什么样的节点会有属于自己的图层？一个是具有层叠上下文属性的节点，一个是节点本身需要被裁剪（典型的，比如 div 中放不下文字，那么文字就需要被“裁剪”，此时文字会产生一个自己的图层）
    # 5）绘制（paint）

    每个图层的绘制依赖于一个个很小的绘制指令，这些绘制指令按顺序组成了一个绘制列表

    接着将绘制列表提交给合成线程，合成线程将图层分为多个图块

    这里之所以要将图层分块处理，是因为视口决定了用户一次可以看到的页面内容是有限的，若一次性将整个图层渲染出来，会对性能有一定影响

    # 6）栅格化（光栅化）

    将图块提交给栅格化线程池，图块经过栅格化转化为位图

    栅格化的过程可能会使用 GPU 加速，最终生成位图的操作实际是在 GPU 中完成的。因此这里可能涉及到了跨进程操作：

    # 7）合成和显示

    一旦所有图块被栅格化成位图，这些位图会返回给合成线程，合成线程再发送一个 DrawQrud 绘制指令给浏览器主进程，浏览器主进程接受该指令，将页面内容绘制到内存中，然后将内存中的页面发送给显卡的后缓冲区，显卡再通过显示器将最终内容显示到屏幕上。

    # 资源引用对 HTML 解析的影响

    在浏览器向服务器发送请求后，服务器会返回 HTML 文件。浏览器对 HTML 文件进行解析（或者说对 DOM 进行解析），构建 DOM 树，在解析 HTML 的过程中，不可避免会遇到 <script> 和 <link>，那么这会对 HTML 的解析产生什么影响呢？

    # 1. 外联 <script> 对 HTML 解析的影响

    外联 <script> 可能会阻塞 DOM 的解析和渲染

    从位置上来看，如果外联 <script> 是放在 </body> 上方，那么其实是不会对 DOM 的解析造成影响的，因为肯定是在 DOM 解析完毕之后再去加载并执行 JS 代码；如果外联 <script> 是放在其它位置，那么根据 <script> 标签不加声明、使用 async 声明或者使用 defer 声明，对 DOM 解析的影响会有所不同。其区别可以用下面这张图形象地表示：

    整个过程主要和浏览器的渲染进程有关，绿色线表示 GUI 渲染线程，负责解析 HTML 和构建 DOM 树的工作，蓝色线表示异步 http 请求线程，负责请求并下载 JS 文件，红色线表示 JS 引擎线程，负责解析并执行 JS 代码。

    • 不加声明：

      在解析 HTML 的过程中，若碰到 <script> 标签，会马上停止解析，由异步 http 请求线程和 JS 引擎线程分别去下载和执行代码，执行完毕之后再由 GUI 渲染线程接手，继续解析 HTML。注意，虽然网络请求和 HTML 解析完全可以并行，但在这里不会这样，在进行网络请求之前，HTML 的解析就已经被阻塞了。

      既然 HTML（DOM）的解析被阻塞了，那么 DOM 的渲染也是被阻塞的。

    • async 声明：

      在解析 HTML 的过程中，若碰到 <script> 标签，异步 http 请求线程会去并行地请求并加载 JS 文件，同时 HTML 仍然继续解析，直到 JS 文件加载完毕，此时 HTML 解析才会停止，交由 JS 引擎线程去执行代码，执行完毕后再继续解析 HTML。在声明 async 的情况下，代码的执行时机取决于 JS 文件加载完成的时机，而文件加载完成的时机通常是不确定的，因此代码的执行时机也是不确定的，顺序引用的 JS 文件在执行上可能是乱序的，这就会导致存在依赖的 JS 文件执行时出现问题。比如 A.js 依赖于 B.js，而它执行的时候 B.js 尚未执行，这时候 A.js 的执行就会出问题。

      同样的，既然 HTML（DOM）的解析被阻塞了，那么 DOM 的渲染也是被阻塞的。

    • defer 声明：

      在解析 HTML 的过程中，若碰到 <script> 标签，异步 http 请求线程会去并行地请求并加载 JS 文件，同时 HTML 仍然继续解析，解析的过程中 JS 文件可能已经加载完毕，但解析完全不会被打断，直到 HTML 解析结束，JS 文件才会被执行。也就是说，defer 声明会在最大程度上保证 HTML 解析过程的顺利执行。

    ==Note：==无论是哪种声明方式，HTML 的解析以及 JS 的执行都不可能同时进行，这不是因为它们是单线程的，而是因为这两个线程是互斥的。

    参考：

    https://javascript.tutorialhorizon.com/2015/08/11/script-async-defer-attribute/

    https://juejin.cn/post/6844903936877395981#heading-0

    # 2. 内联 <script> 对 HTML 解析的影响

    内联 <script> 一定会阻塞 DOM 的解析和渲染

    前面说的是外联 <script> 的情况，所谓外联那必然就要请求 JS 文件，所以不声明 async 和 defer、只声明 async 以及只声明 defer 是有区别的；但如果是内联的 <script>，那么就不存在请求 JS 文件的说法，async 和 defer 的属性声明也就不生效了。

    内联 <script> 的情况简单很多，只要遇到 <script> 标签，HTML 的解析就会停止，先去执行 JS 代码，执行完毕后再继续解析 HTML。

    # 3. <link> 对 DOM 解析和渲染的影响

    <link> 不会阻塞 DOM 的解析，但会阻塞 DOM 的渲染

    实际上，可以认为 GUI 渲染线程会并行做两件事，一个是解析 HTML（解析 DOM），构建 DOM 树；一个是解析 CSS，构建 CSSOM 树，两者互不影响。因此，<link> 并不会阻塞 DOM 的解析，一旦遇到 <link>，只需要让异步 http 请求线程去请求并加载 CSS 文件，之后对 CSS 解析即可，与此同时，DOM 的解析仍可正常继续进行。

    但是，页面最终能够渲染出来，依靠的是 DOM 树和 CSSOM 树合并形成的 render 树（layout 树），因此，<link> 是会阻塞 DOM（或者说页面）的渲染的。

    # 4. CSS 对 JS 执行的影响

    此外，解析 <link> （或者说 CSS）的时候是会阻塞 JS 执行的。一方面，由于 CSS 的解析在 GUI 渲染线程上进行，JS 的执行在 JS 引擎线程上进行，这两个线程互斥，因此解析 <link> 与执行 JS 不可能同时进行；另一方面，若二者同时进行，JS 可能会访问当时尚未解析出来的样式信息，导致执行上出错，从这个角度说，在设计上也不应该让他们两者可以同时进行。

    # 5. 如何理解阻塞 DOM 的渲染？

    如果说阻塞 DOM（页面）的渲染就是让页面一片空白，什么也没有的话，那么用户体验是非常差的。所以，这里的阻塞指的是阻塞后续 DOM 的渲染，遇到 <script> 之前的 DOM 渲染不仅不会阻塞，还会基于它们触发一次 paint，提前将这部分内容显示出来。但不是什么 <script> 都会触发 paint，下面是例外：

    • <head> 中的 <script> 不会触发，因为此时甚至连 <body> 都还没解析，触发了也没有什么内容可以显示
    • 内联的 <script> 不会触发

    # 重排和重绘

    # 重排

    重排即 reflow，也叫回流。如果通过 CSS 或者 JS 修改某个元素的几何位置属性，比如修改了宽高等，那么其它元素的布局也可能跟着改变，这会触发浏览器的重新布局，从 style（计算样式）这一步开始将整个渲染流水线重新走一遍（如果是修改了 DOM 结构，那么 DOM 树也需要重新构建）。因此，重排的开销是非常大的。

    重排发生的关键在于布局的改变。以下场景会引起重排：

    • 页面渲染初始化
    • 添加或者删除可见的 DOM 元素
    • 元素位置改变
    • 元素大小改变（包括宽高、内外边距、边框）
    • 元素的 display 改变（none 和 block 切换）
    • 填充内容改变，比如修改文本、修改图片大小等
    • 激活伪类（鼠标移入移出等）
    • 浏览器窗口尺寸改变（触发 resize 事件）
    • 读取某些样式属性（和浏览器的渲染队列有关，后面解释）

    PS：设置了绝对定位或者固定定位的元素脱离了正常文档流，修改 top 等值只会引起自身重排，不会影响到其它元素，因此重排的开销相对来说比较小。（注意不是不发生重排，毕竟元素的位置是改变了的）

    # 重绘

    重绘即 repaint。比如修改了某个元素的背景颜色，那么会从 paint（绘制）这一步开始将整个渲染流水线重新走一遍。注意重绘并不会重新走一遍 layout 和 layer，因为我们只是修改了颜色，浏览器只需要重新“绘画”，布局和分层不会受到影响。

    从图中可以看出，重排是包含重绘的，发生了重排，那么也一定会发生重绘，而发生了重绘则不一定会发生重排。虽然重绘也有一定的性能开销，但因为不需要重新进行布局和分层，所以执行效率要比重排高。

    重绘发生的关键在于外观的改变。以下场景会引起重绘：

    • 所有会引起重排的操作
    • 修改下面这些属性：
      • color、visibility、text-decoration、font-family、opacity（除非将元素提升到合成层）
      • background、background-image、background-position、background-repeat、background-size
      • outline、outline-color、outline-style、outline-width
      • border-radius、border-style、box-shadow
      • 具体的属性可以到 csstrigger (opens new window) 查看，注意浏览器之间的差别

    # 渲染队列

    前面说过读取某些样式属性的时候会触发重排 + 重绘，明明只是读取属性而没有修改属性，为什么还会引起重排和重绘呢？这需要从浏览器的渲染队列讲起

    看下面这段代码：

    div.style.left = '10px';
div.style.top = '10px';
div.style.width = '20px';
div.style.height = '20px';

    根据前面的分析，这段代码应该引起四次重排，但由于渲染队列的存在，实际上只引起一次重排。

    从优化的角度出发，浏览器不会每执行一句代码就进行一次重排，而是先把这些会引起重排或者重绘的操作放到一个渲染队列中，等到队列中的操作达到一定数量或者到了一定的时间间隔之后，再 flush （清空）队列，批量执行这些操作，这样就将原本的四次重排压缩到了一次。

    强制 / 手动 flush 队列：

    当我们访问下面这些样式属性的时候：

    • offsetTop, offsetLeft, offsetWidth, offsetHeight
    • scrollTop, scrollLeft, scrollWidth, scrollHeight
    • clientTop, clientLeft, clientWidth, clientHeight
    • getComputedStyle(), 或者 IE 的 currentStyle

    我们肯定希望得到最新的属性值，而渲染队列中的操作很可能会影响到这些属性值，因此，浏览器为了确保我们可以得到最新的属性值，会先去 flush 一次队列，将队列里的操作马上批量执行一遍，以对属性值进行可能的更新。而这次 flush 就会触发一次重排 + 重绘。

    再看修改之后的代码：

    div.style.left = '10px';
console.log(div.offsetLeft);
div.style.top = '10px';
console.log(div.offsetTop);
div.style.width = '20px';
console.log(div.offsetWidth);
div.style.height = '20px';
console.log(div.offsetHeight);

    每次修改属性之后都即时对属性进行了访问，虽然修改属性之后不会马上触发重排，但是紧接着的访问操作会 flush 队列并触发重排。这段代码总共会触发四次重排。

    Note：尽量避免频繁访问这些属性，每次访问可能都意味着一次重排

    # 性能优化：如何减少重排和重绘

    既然重排和重绘都会造成一定的性能开销，那么如何减少重排和重绘呢？

    # 1）分离读写操作

    拿这段代码举例：

    div.style.left = '10px';
console.log(div.offsetLeft);
div.style.top = '10px';
console.log(div.offsetTop);
div.style.width = '20px';
console.log(div.offsetWidth);
div.style.height = '20px';
console.log(div.offsetHeight);

    它是一写一读的，每次读的时候都会刷新渲染队列，把上次的写操作实际执行，从而触发重排。

    但是如果改成下面这种方式：

    div.style.left = '10px';
div.style.top = '10px';
div.style.width = '20px';
div.style.height = '20px';
console.log(div.offsetLeft);
console.log(div.offsetTop);
console.log(div.offsetWidth);
console.log(div.offsetHeight);

    那么前四次写操作可以依次进入渲染队列，之后遇到第一次读操作的时候发生队列刷新，执行所有写操作，整个过程只触发了一次重排。

    分离读写操作的另外一种形式是利用变量缓存布局信息：

    假设当前渲染队列非空，那么这段代码将会触发两次重排：

    div.style.left = div.offsetLeft + 1 + 'px';
div.style.top = div.offsetTop + 1 + 'px';

    而如果是这段代码：

    var curLeft = div.offsetLeft;
var curTop = div.offsetTop;
div.style.left = curLeft + 1 + 'px';
div.style.top = curTop + 1 + 'px';

    那么相当于是将读操作的结果寄放在变量里，从而实现读写分离。这样，我们仅在第一次通过 div.offsetLeft 进行读操作的时候会触发一次重排。

    # 2）样式集中一次修改

    现代浏览器大多支持渲染队列机制，可以保证连续的写操作不会触发连续的重排，但是旧的浏览器怎么办呢？我们通常会想办法集中一次修改样式。

    比如使用 cssText：

    el.style.cssText = "left:10px;top:10px;width:20px;height:20px"

    或者给元素添加一个类：

    new_class {
    left: 10px;
    top: 10px;
    width: 20px;
    height: 20px;
}

el.classList.add("new_class")
    # 3）脱离文档的 DOM 树（布局树）执行操作

    既然修改直接来自 DOM 树中的节点可能会引发重排重绘，那么我们可以想办法让节点以某种形式”暂时脱离原有的 DOM 树“，执行完操作之后，再让它“回归到 DOM 树中”。以下有三种方式都是基于这个原理的。

    方式一：隐藏元素 => 执行操作 => 显示元素：

    前面说过，布局树中只会包含最终可见的元素，因此只要给元素设置 display: none，那么它就会脱离布局树，修改它的任何属性也自然不会引起页面的重排和重绘。在修改完之后，我们只要让元素重新显示即可。

    el.style.display = "none"
el.style.width = "10px"
el.style.display = "block"

    注意：这种方式只是避免了两次 display 之间的属性修改操作带来的重排和重绘，但每次 display 的时候还是会触发一次重排。总的来说，它可以将重排的次数压缩到两次。

    这里的 display 如果改为使用 visibility 会怎么样？

    el.style.visibility  = "hidden"
el.style.width = "10px"
el.style.visibility = "visible"

    设置了 visibility: hidden 的元素仍然占据页面空间，也没有从布局树中消失，只是视觉上看不见而已。因此在设置 visibility 的时候不会触发重排，只会触发重绘。而正由于它没有从布局树中消失，所以两次 visibility 之间的属性修改操作是可能会带来重排和重绘的。

    也就是说，相比 display，visibility 将两次重排优化成了两次重绘，但是中间由于元素本身修改导致的重排重绘将无法避免。

    方式二：document.createDocumentFragment() 创建文档碎片：

    场景：现在要往一个 ul#list 中插入多个 li。

    如果是下面的代码：

    const list = document.querySelector('#list');
const fruits = ['Apple','Orange','Banana','Melon']
fruits.forEach(fruit => {
    const li = document.createElement('li');
    li.innerHTML = fruit;
    list.appendChild(li);
});

    那么每一次循环都会执行一次添加 DOM 的操作，对于不支持渲染队列机制的旧浏览器，这些连续的操作将会触发四次重排（前面说过，添加 DOM 会触发重排）。

    而文档碎片的原理是，将所有这些添加 DOM 的操作集中在文档碎片中进行，由于碎片是一棵独立于正常文档流 DOM 树的树，因此这些操作不会对原有页面产生影响，更不会导致重排。

    具体代码如下：

    const list = document.querySelector('#list');
const fruits = ['Apple', 'Orange', 'Banana', 'Melon'];

const fragment = document.createDocumentFragment();

fruits.forEach(fruit => {
  const li = document.createElement('li');
  li.innerHTML = fruit;
  fragment.appendChild(li);
});

list.appendChild(fragment);

    最后再将碎片 DOM 一次性加入到文档中即可。这样，我们将重排次数压缩至一次。

    方式三：拷贝节点 => 执行操作 => 替换节点：

    调用 cloneNode() 拷贝得到的节点独立于正常文档流的 DOM 树，对他执行的操作不会引起重排。因此我们可以先拷贝原节点得到副本，在外围对副本执行完操作之后再用它替换原节点。

    const el2 = el1.cloneNode(true)
el2.style.width = "10px"
el1.parentNode.replaceChild(el2,el2)
    # 4）设置元素为绝对定位或固定定位

    前面说过，元素设置为绝对定位或者固定定位之后就会脱离文档流，对元素本身属性的修改虽然仍会引起重排/重绘，但这种重排/重绘只发生在自身，不会影响到正常文档流中的元素。

    # 5）开启 GPU 硬件加速

    GPU（图像处理器） 硬件加速是指应用 GPU 的图形性能处理浏览器中的一些图形操作，因为 GPU 是专门为处理图形而设计，所以它在速度和能耗上更有效率。

    应用了 transition 或者 animation 的元素，其所在的层将由渲染层提升为合成层，此后即使修改 opacity 或者 transform，也不会触发重排和重绘。从下图可以看出，这样的元素会跳过渲染进程主线程的布局、分层和绘制阶段，直接来到非主线程的合成阶段，由 GPU 接管渲染工作。

    CSS3 通常依靠 transition 或者 animation 来做动画，在 GPU 硬件加速的加持下，CSS3 动画的效率比较高。当然也不是说 CSS3 动画的使用是无节制的，滥用动画意味着滥用硬件加速，还是会消耗一定内存的。

    那么如果我们想主动开启 GPU 加速，应该怎么做呢？

    第一种方法是使用 will-change 属性：

    div {
    will-change: opacity transform
}

    这相当于是告诉渲染引擎，接下来 div 要在 opacity 和 transform 上进行变换，这会使得 div 从渲染层提升到合成层，从而在这两个属性改变时由 GPU 进行渲染，避开重排和重绘。

    对于不支持 will-change 属性的浏览器，还可以使用下面的 hack 方式开启 GPU 加速：

    div {
    transform: translateZ(0)
}
